Argo Events + Argo Workflows 웹훅 테스트 구성 학습 정리

작성 기준: 2026-06-10, repo: C:\Users\user\sungju-argocd-test-repo

목표

외부에서 웹훅 메시지를 보내면 Argo Events가 요청을 받고, Sensor가 Argo Workflow를 생성하며, Workflow가 받은 메시지를 테스트용 웹 서비스로 그대로 전달한다. 테스트용 웹 서비스는 받은 요청을 메모리에 저장하고 브라우저에서 최근 요청을 확인할 수 있게 한다.

1. 전체 구성 요약

이번 repo에는 Kubernetes 리소스를 선언한 YAML만 올라간다. ArgoCD가 이 repo를 sync하면 클러스터에 아래 리소스가 생성된다.

파일생성되는 리소스역할
argoevent/00-namespace.yaml Namespace 테스트 리소스를 argo-webhook-test 네임스페이스에 모은다.
argoevent/10-webhook-receiver.yaml ConfigMap, Deployment, Service 요청 저장/조회용 Python 웹 서비스와 NodePort 31080을 만든다.
argoevent/20-argo-events-workflow.yaml EventBus, ServiceAccount, Role, RoleBinding, WorkflowTemplate, EventSource, Service, Sensor 웹훅 수신, 이벤트 전달, Workflow 생성, receiver로 POST 전달을 구성한다.

외부 접속 포트

용도URL설명
웹훅 입력 http://10.0.0.200:31081/webhook 외부에서 POST 요청을 보내는 Argo Events EventSource 엔드포인트
결과 확인 UI http://10.0.0.200:31080/ Workflow가 receiver로 보낸 요청을 브라우저에서 확인하는 페이지

요청 흐름

사용자 curl
  -> NodePort 31081
  -> EventSource(webhook-eventsource)
  -> EventBus(default, NATS)
  -> Sensor(webhook-forwarder)
  -> Workflow 생성(forward-webhook-xxxxx)
  -> curl 컨테이너 실행
  -> ClusterIP Service webhook-receiver
  -> receiver Python 앱 메모리에 저장
  -> NodePort 31080에서 브라우저로 확인

2. Namespace YAML

모든 테스트 리소스를 한 네임스페이스에 모으면 삭제, 조회, 권한 관리가 쉬워진다.

apiVersion: v1
kind: Namespace
metadata:
  name: argo-webhook-test
필드설명
apiVersion: v1Kubernetes 기본 리소스 API 그룹이다.
kind: Namespace네임스페이스를 생성한다.
metadata.name이후 모든 리소스의 namespace 값과 맞아야 한다.

3. 웹훅 테스트용 웹 서비스

10-webhook-receiver.yaml은 요청을 저장하고 화면에 보여주는 테스트 웹 서비스를 만든다. 데이터베이스 없이 프로세스 메모리에 저장하므로 Pod가 재시작되면 저장된 요청은 사라진다. 테스트용으로는 단순하고 빠르다는 장점이 있다.

3.1 ConfigMap

Python 애플리케이션 코드를 컨테이너 이미지에 굽지 않고 ConfigMap으로 주입한다.

apiVersion: v1
kind: ConfigMap
metadata:
  name: webhook-receiver-app
  namespace: argo-webhook-test
data:
  app.py: |
    ... Python HTTP server code ...
필드설명
data.app.py컨테이너 안에서 /app/app.py로 마운트될 Python 코드다.
EVENTS = []받은 요청을 저장하는 메모리 리스트다.
MAX_EVENTS = 100최대 100건만 보관한다.
do_POSTPOST 요청 body, header, path, method, time을 저장한다.
do_GET/에서는 HTML 화면, /api/events에서는 JSON 배열을 반환한다.

3.2 Deployment

receiver 앱을 실제 Pod로 실행한다.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: webhook-receiver
  namespace: argo-webhook-test
spec:
  replicas: 1
  selector:
    matchLabels:
      app.kubernetes.io/name: webhook-receiver
  template:
    metadata:
      labels:
        app.kubernetes.io/name: webhook-receiver
    spec:
      containers:
        - name: receiver
          image: python:3.12-alpine
          command: ["python", "/app/app.py"]
          ports:
            - name: http
              containerPort: 8080
          volumeMounts:
            - name: app
              mountPath: /app
              readOnly: true
      volumes:
        - name: app
          configMap:
            name: webhook-receiver-app
필드설명
replicas: 1메모리에 요청을 저장하므로 replica를 1개로 둔다. 2개 이상이면 요청이 Pod별로 나뉘어 저장된다.
selector.matchLabelsDeployment가 관리할 Pod를 고르는 label이다.
image: python:3.12-alpine별도 빌드 없이 Python 기본 이미지로 실행한다.
commandConfigMap에서 마운트된 /app/app.py를 실행한다.
containerPort: 8080Python 서버가 Pod 내부에서 listen하는 포트다.
volumeMountsConfigMap의 app.py를 컨테이너의 /app 디렉터리에 읽기 전용으로 붙인다.

3.3 Service(NodePort)

브라우저에서 receiver UI를 볼 수 있도록 NodePort를 열었다.

apiVersion: v1
kind: Service
metadata:
  name: webhook-receiver
  namespace: argo-webhook-test
spec:
  type: NodePort
  selector:
    app.kubernetes.io/name: webhook-receiver
  ports:
    - name: http
      port: 80
      targetPort: http
      nodePort: 31080
필드설명
type: NodePort클러스터 노드 IP와 고정 포트로 외부 접속을 허용한다.
selectorDeployment Pod의 label과 맞아야 트래픽이 Pod로 간다.
port: 80클러스터 내부 서비스 포트다. Workflow에서는 http://webhook-receiver...로 접근한다.
targetPort: httpPod의 named port http, 즉 8080으로 연결한다.
nodePort: 31080외부 접속 포트다. 이미 사용 중이면 ArgoCD sync가 실패하므로 다른 값으로 바꿔야 한다.

4. Argo Events + Workflow 파이프라인

20-argo-events-workflow.yaml은 Argo Events가 웹훅을 받고 Workflow를 실행하는 전체 파이프라인이다.

4.1 EventBus

EventSource와 Sensor 사이에서 이벤트를 전달하는 버스다. 여기서는 테스트 목적이라 네이티브 NATS 1개 replica를 사용했다.

apiVersion: argoproj.io/v1alpha1
kind: EventBus
metadata:
  name: default
  namespace: argo-webhook-test
spec:
  nats:
    native:
      replicas: 1
필드설명
kind: EventBusArgo Events의 이벤트 전달 인프라다.
metadata.name: defaultEventSource와 Sensor의 eventBusName: default와 연결된다.
nats.native.replicas: 1테스트용 단일 NATS다. 운영 고가용성이 필요하면 replica와 구성을 조정한다.

4.2 ServiceAccount, Role, RoleBinding

Sensor는 이벤트를 받으면 Kubernetes API를 호출해서 Workflow 리소스를 생성해야 한다. 이를 위해 권한을 부여한다.

apiVersion: v1
kind: ServiceAccount
metadata:
  name: webhook-sensor
  namespace: argo-webhook-test
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: webhook-sensor
  namespace: argo-webhook-test
rules:
  - apiGroups: ["argoproj.io"]
    resources: ["workflows"]
    verbs: ["create", "get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: webhook-sensor
  namespace: argo-webhook-test
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: webhook-sensor
subjects:
  - kind: ServiceAccount
    name: webhook-sensor
    namespace: argo-webhook-test
리소스역할
ServiceAccountSensor Pod가 사용할 Kubernetes 계정이다.
Roleargoproj.io API 그룹의 workflows에 대해 생성/조회 권한을 준다.
RoleBinding위 Role을 webhook-sensor ServiceAccount에 연결한다.

4.3 WorkflowTemplate

실제 전달 작업을 수행하는 Workflow의 템플릿이다. Sensor는 매번 새 Workflow를 만들고, 그 Workflow가 이 템플릿을 참조한다.

apiVersion: argoproj.io/v1alpha1
kind: WorkflowTemplate
metadata:
  name: forward-webhook-to-receiver
  namespace: argo-webhook-test
spec:
  entrypoint: forward
  arguments:
    parameters:
      - name: payload
        value: "{}"
  templates:
    - name: forward
      inputs:
        parameters:
          - name: payload
      container:
        image: curlimages/curl:8.10.1
        command: [sh, -c]
        args:
          - |
            set -eu
            printf '%s' "$PAYLOAD" > /tmp/payload
            curl -sS -X POST "$TARGET_URL" \
              -H "Content-Type: application/json" \
              --data-binary @/tmp/payload
        env:
          - name: TARGET_URL
            value: http://webhook-receiver.argo-webhook-test.svc.cluster.local/
          - name: PAYLOAD
            value: "{{inputs.parameters.payload}}"
필드설명
entrypoint: forwardWorkflow가 시작할 template 이름이다.
arguments.parameters.payloadSensor가 웹훅 body를 넣어줄 파라미터다.
image: curlimages/curl:8.10.1HTTP POST를 보내기 위한 작은 curl 이미지다.
printf '%s' "$PAYLOAD" > /tmp/payloadpayload를 파일로 저장한다. shell quoting 문제를 줄이고 body를 그대로 보내기 위한 방식이다.
--data-binary @/tmp/payload파일 내용을 HTTP body로 전송한다.
TARGET_URLreceiver의 클러스터 내부 DNS 주소다. NodePort가 아니라 ClusterIP Service로 통신한다.

4.4 EventSource

외부 웹훅 요청을 받는 Argo Events 리소스다. /webhook 경로로 POST 요청을 받는다.

apiVersion: argoproj.io/v1alpha1
kind: EventSource
metadata:
  name: webhook-eventsource
  namespace: argo-webhook-test
spec:
  eventBusName: default
  service:
    ports:
      - name: webhook
        port: 12000
        targetPort: 12000
  webhook:
    message:
      endpoint: /webhook
      jsonBody: false
      method: POST
      port: "12000"
필드설명
eventBusName: default수신한 이벤트를 default EventBus로 보낸다.
service.portsArgo Events가 EventSource Pod 앞에 내부 Service를 만들 때 사용할 포트다.
webhook.messagemessage라는 이벤트 이름을 정의한다. Sensor의 eventName과 일치해야 한다.
endpoint: /webhook외부에서 호출할 path다.
method: POSTPOST만 이벤트로 받는다.
jsonBody: falseJSON을 객체로 파싱하지 않고 raw body 문자열로 보관한다. “메시지 그대로 전달”하려고 넣은 설정이다.

4.5 EventSource 외부 노출 Service

EventSource에 접근할 수 있도록 별도의 NodePort Service를 만들었다. Argo Events가 내부 Service를 만들 수 있지만, 여기서는 외부 접속 포트를 명확히 고정하기 위해 직접 Service를 추가했다.

apiVersion: v1
kind: Service
metadata:
  name: webhook-eventsource-nodeport
  namespace: argo-webhook-test
spec:
  type: NodePort
  selector:
    eventsource-name: webhook-eventsource
  ports:
    - name: webhook
      port: 12000
      targetPort: 12000
      nodePort: 31081
필드설명
selector.eventsource-nameArgo Events EventSource Pod에 붙는 label을 선택한다.
nodePort: 31081외부 웹훅 입력 포트다.
port/targetPort: 12000EventSource가 listen하는 내부 포트다.

4.6 Sensor

Sensor는 EventSource 이벤트를 감지하고, 그 이벤트의 body를 Workflow 파라미터로 넣어서 Workflow를 생성한다.

apiVersion: argoproj.io/v1alpha1
kind: Sensor
metadata:
  name: webhook-forwarder
  namespace: argo-webhook-test
spec:
  eventBusName: default
  template:
    serviceAccountName: webhook-sensor
  dependencies:
    - name: inbound-webhook
      eventSourceName: webhook-eventsource
      eventName: message
  triggers:
    - template:
        name: forward-to-receiver
        k8s:
          operation: create
          source:
            resource:
              apiVersion: argoproj.io/v1alpha1
              kind: Workflow
              metadata:
                generateName: forward-webhook-
                namespace: argo-webhook-test
              spec:
                workflowTemplateRef:
                  name: forward-webhook-to-receiver
                arguments:
                  parameters:
                    - name: payload
                      value: "{}"
          parameters:
            - src:
                dependencyName: inbound-webhook
                dataKey: body
              dest: spec.arguments.parameters.0.value
필드설명
template.serviceAccountNameWorkflow 생성 권한을 가진 ServiceAccount를 Sensor에 적용한다.
dependencies어떤 EventSource의 어떤 이벤트를 기다릴지 정의한다.
eventSourceNamewebhook-eventsource와 연결된다.
eventNameEventSource의 webhook.message에서 message와 맞아야 한다.
triggers.template.k8s.operation: create이벤트가 오면 Kubernetes 리소스를 생성한다.
kind: Workflow생성 대상은 Argo Workflow다.
generateName: forward-webhook-Workflow 이름을 매번 자동 생성한다.
workflowTemplateRef앞에서 만든 WorkflowTemplate을 참조한다.
dataKey: bodyEventSource 이벤트 payload에서 body 값을 읽는다.
dest: spec.arguments.parameters.0.value읽은 body를 생성될 Workflow의 첫 번째 파라미터 값에 넣는다.

5. 사용 방법

5.1 ArgoCD로 배포

ArgoCD Application의 path를 repo의 argoevent로 잡고 sync한다. 전체 repo root를 path로 잡아도 YAML은 argoevent 아래에만 있으므로 동작할 수 있지만, 명확하게 하려면 path를 argoevent로 지정하는 것이 좋다.

repo: sungju-argocd-test-repo
path: argoevent
namespace: argo-webhook-test 또는 cluster default 설정
sync: manual 또는 automatic

Argo Events CRD와 Argo Workflows CRD는 클러스터에 이미 설치되어 있어야 한다. 설치되어 있지 않으면 EventBus, EventSource, Sensor, WorkflowTemplate 같은 리소스를 Kubernetes가 알 수 없어 sync가 실패한다.

5.2 웹훅 전송

Windows PowerShell에서는 아래처럼 요청을 보낸다.

curl.exe -X POST http://10.0.0.200:31081/webhook `
  -H "Content-Type: application/json" `
  -d "{\"hello\":\"argo\"}"

Linux/macOS shell에서는 아래처럼 보낼 수 있다.

curl -X POST http://10.0.0.200:31081/webhook \
  -H "Content-Type: application/json" \
  -d '{"hello":"argo"}'

5.3 결과 확인

브라우저에서 아래 주소를 연다.

http://10.0.0.200:31080/

JSON API로 보고 싶으면 아래 주소를 사용한다.

http://10.0.0.200:31080/api/events

5.4 정상 동작 시 보이는 것

  1. curl 요청이 31081로 들어간다.
  2. Argo Events EventSource Pod가 이벤트를 만든다.
  3. Sensor가 forward-webhook-xxxxx Workflow를 생성한다.
  4. Workflow Pod가 curl 컨테이너로 receiver에 POST한다.
  5. 31080 페이지에 요청 body와 header가 표시된다.

7. 문제 해결

7.1 NodePort already allocated

이미 다른 Service가 같은 NodePort를 쓰고 있으면 sync가 실패한다.

Service "webhook-receiver" is invalid:
spec.ports[0].nodePort: Invalid value: 31080: provided port is already allocated

해결 방법은 nodePort 값을 Kubernetes NodePort 범위 안의 사용 가능한 값으로 바꾸는 것이다. 기본 범위는 보통 30000-32767이다.

# receiver UI 포트 변경
argoevent/10-webhook-receiver.yaml
nodePort: 31080

# webhook 입력 포트 변경
argoevent/20-argo-events-workflow.yaml
nodePort: 31081

7.2 웹훅을 보냈는데 화면에 안 뜸

  1. curl 요청 주소가 31081/webhook인지 확인한다.
  2. ArgoCD sync가 완료됐는지 확인한다.
  3. EventSource, Sensor Pod가 떠 있는지 확인한다.
  4. Workflow가 생성됐는지 확인한다.
  5. Workflow Pod 로그에서 curl 실패가 있는지 확인한다.
  6. receiver Service DNS가 맞는지 확인한다: webhook-receiver.argo-webhook-test.svc.cluster.local

7.3 Workflow가 안 만들어짐

Sensor 권한 또는 dependency 연결 문제일 수 있다.

kubectl describe sensor webhook-forwarder -n argo-webhook-test
kubectl get role,rolebinding,serviceaccount -n argo-webhook-test

특히 아래 값들이 서로 맞아야 한다.

  • EventSource 이름: webhook-eventsource
  • EventSource 이벤트 이름: message
  • Sensor dependency의 eventSourceName: webhook-eventsource
  • Sensor dependency의 eventName: message
  • Sensor ServiceAccount: webhook-sensor

7.4 JSON이 이상하게 바뀜

이번 YAML에는 jsonBody: false를 넣었다. 이 설정은 EventSource가 요청 body를 JSON 객체로 파싱하지 않고 문자열로 유지하게 한다. 메시지를 최대한 그대로 전달하고 싶을 때 중요하다.

8. 수정 포인트별 가이드

receiver UI 포트를 바꾸고 싶을 때

argoevent/10-webhook-receiver.yaml에서 Service의 nodePort를 바꾼다. README의 확인 URL도 같이 바꾼다.

nodePort: 31080

webhook 입력 포트를 바꾸고 싶을 때

argoevent/20-argo-events-workflow.yaml의 webhook-eventsource-nodeport Service에서 nodePort를 바꾼다. README의 curl URL도 같이 바꾼다.

nodePort: 31081

webhook path를 바꾸고 싶을 때

EventSource의 endpoint를 바꾼다.

webhook:
  message:
    endpoint: /webhook

전달 대상 서비스를 바꾸고 싶을 때

WorkflowTemplate의 TARGET_URL을 바꾼다.

env:
  - name: TARGET_URL
    value: http://webhook-receiver.argo-webhook-test.svc.cluster.local/

받은 요청을 더 오래 저장하고 싶을 때

현재 receiver는 Pod 메모리에만 저장한다. 오래 저장하려면 다음 중 하나로 확장한다.

  • 간단한 테스트: MAX_EVENTS 값을 늘린다.
  • Pod 재시작 후에도 보존: PVC에 파일로 저장한다.
  • 운영식 저장: PostgreSQL, Redis, Elasticsearch 같은 외부 저장소를 붙인다.

9. 학습 포인트 정리

  • EventSource는 외부 이벤트를 Kubernetes 안으로 가져오는 입구다.
  • EventBus는 EventSource와 Sensor 사이에서 이벤트를 운반한다.
  • Sensor는 이벤트를 보고 어떤 액션을 할지 결정한다.
  • WorkflowTemplate은 반복 실행할 Workflow 로직을 템플릿으로 정의한다.
  • Workflow는 실제 실행 인스턴스다. 이번 구성에서는 웹훅 요청 1건마다 Workflow 1개가 만들어진다.
  • ServiceAccount + Role + RoleBinding은 Sensor가 Workflow를 생성할 권한을 주기 위해 필요하다.
  • NodePort는 외부에서 노드 IP와 포트로 접속하기 위한 Service 타입이다.
  • 클러스터 내부 통신은 NodePort가 아니라 Service DNS를 쓰는 것이 일반적이다.
